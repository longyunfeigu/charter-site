Charter 始于一个简单的问题：最强的编程 Agent 都在命令行里， 好用的 IDE 却往往绑定自家 Agent。 为什么非得在“信任的 Agent”和“顺手的 IDE”之间二选一？

从 Prompt 到 Proof，看清 Agent 的每一步

  • Agent CLI

    最前沿的 Agent，都在终端里

    Claude Code 和 Codex 代表着当下最前沿的能力。但在终端里， 你只能看着输出一屏屏滚过——没有 Diff 界面、没有审查关卡，也无法回放。

  • Charter

    它们的 IDE，你的规则

    给你信任的 Agent 配上一套真正的 IDE：编辑器、Worktree、真终端， 再把证据留痕、审查关卡和回滚补齐。终端的优势也都保留—— 持久 PTY、清晰的命令块、多 Session 并行。

  • AI-first IDE

    好用的 IDE，换不了的 Agent

    IDE 做得再漂亮，内置 Agent 也已经由厂商替你选好。 编辑器是你的，敲键盘的那双手却不是。

Session = project+ agent+ worktree+ conversation+ plan+ files+ terminals+ preview+ verification+ review

Session 是人和 Agent 协作的完整单元——一个任务需要的上下文都收在这里，不会散落各处。

一次委托，三个关键时刻

下面的画面都来自同一个真实 Session： 选 Agent、看过程、验结果。

01 · 委托

Agent 随你选，工作流不用换

使用内置的 Charter Agent，也可以原生运行你已经安装的 Claude Code 或 Codex CLI。不管选谁，都沿用 同一套 Session 和审查流程，不必来回切换工具。

每个任务给多少权限，由你决定：Read、 Approve、Auto 或 Full——从只读问答，到放手让它干。

02 · 观察

证据就在对话旁边

Agent 运行时，文件写入、命令、Diff、预览、检查和决策都会按时间顺序 记进同一本账本；时间戳和增删统计会实时更新。

不用再去聊天记录里考古：工作日志就铺在对话旁边； Session 结束后，还能按步骤回放。

03 · 验收

先看证据，再决定是否批准

Agent 说“完成”，只是把任务交到你手上，等待审查。 审查页展示改动文件、验证历史和最终结果，而不是让模型用一段总结自说自话。

看过带行号的 Diff、重新运行已经记录的检查后，再决定请求修改、回滚或批准； 既可整单处理，也可细到单个文件或改动块。

敢放手，因为随时收得回

多数工具只管把 Agent 跑起来。Charter 更关心你按下回车之后的事—— 从输入 Prompt 到实际改动代码，中间要过四道关，每一步都有记录。

  1. 01

    计划

    动手之前，Agent 先说明步骤、影响范围和验证方式。 你可以批准、修改或驳回。

  2. 02

    权限

    每次工具调用都要先过统一网关。危险操作必须先问你； 禁止的操作则会被直接拦下。

  3. 03

    验证

    验证命令会在真实 Worktree 中运行，结果完整留存； 即使重跑，也不会覆盖之前的失败记录。

  4. 04

    审查

    Agent 完成后，任务只会进入待审查，不会自动接受。 即使已经接受，仍可按字节精确回滚。

即使主动开启 Full 模式，网关、审计记录和硬性限制依然保留—— 验证失败会自动退回审查，已经接受的改动也照样能回滚。

权限分级，不是一键全开

操作越危险，限制越严格。到了最危险的 R4，不是弹窗警告，而是直接禁止。

R0 读取 读文件、搜索、诊断、git status 允许
R1 工作区写入 在隔离 Worktree 内创建或编辑文件 询问确认，或按计划和模式放行
R2 本地执行 已知的本地命令与验证检查 已知检查直接运行；未知命令先询问
R3 联网 / 不可逆 联网或可能造成外部影响的操作 每次都要明确确认
R4 硬性禁止 sudo、git push、读取密钥、向工作区外写入 Charter 直接拒绝

R4 没有“强制运行”这个选项。

模型无法直接接触你的文件

模型运行在独立进程中，无法访问文件系统、Shell 和密钥。 所有工具请求都会经过同一个可审计网关：校验参数、确认边界、 执行权限策略，并留下证据。

  • 隔离写在架构里

    界面、沙箱桥接层、桌面主进程和模型进程各有独立的信任边界。 真正执行工具的是桌面主进程，模型进程从不直接执行工具。

  • 改动默认可回滚

    每次改动前都会创建快照；写入采用原子操作，并校验版本。 回滚可以逐字节恢复文件，还会逐个核对哈希。

  • 密钥待在钥匙串里

    Provider 密钥交给操作系统的加密存储保管，不会传到界面层； 界面只知道这把密钥是否存在。

  • 本地优先，去向说清

    项目、任务状态和证据都保存在本机。只有 Prompt 和你附加的上下文 会发往你配置的模型端点，并受对应 Provider 的数据政策约束。

应用层权限不等于操作系统沙箱。批准命令前请仔细检查； 处理不可信仓库时，仍应使用隔离环境。

门背后，是一个完整的 IDE

四道关是 Charter 的核心，但日常工具一件没少—— 这同样是一套能认真干活的 IDE，不是套着文件树的聊天窗口。

  • 完成通知

    任何 Session 一结束，右上角就会滑入通知：成功为绿色、待审为琥珀色、 失败为红色。对应的 Session 会在侧栏持续闪动，直到你打开。

  • 动静看得见

    Agent 正在写哪个文件，对应卡片就会显示动态效果； 呼吸灯让你一眼看出工作进行到哪里。

  • 真终端

    持久 PTY，每条命令按块展示，长任务结束时还会主动提醒。 切换 Session 后，PTY 仍会继续运行。

  • Session 回放

    沿着时间轴回放已结束的 Session，看清每次改动如何落地。 全程只读，不会改动你的 Worktree。

  • 技能库

    把可复用的工作流集中放进技能库， 在 Composer 中输入 / 即可调用。

  • 随你调校

    四套统一设计的主题，均支持浅色和深色；还有命令面板和快速启动器。 数据统计默认关闭。

  • 真正的编辑器

    Monaco 支持分屏、查找替换，大文件也能稳妥处理。 Agent 在旁边改代码，也不会抢走你的光标。

  • 内置 Git

    状态、Diff、暂存、提交、分支，一应俱全。 按设计，Agent 永远不能运行 git push。

  • 全仓搜索

    以 ripgrep 级的速度搜索整个仓库；执行批量替换前， 先让你预览所有改动。

  • 代码智能

    完整支持 TypeScript 和 JavaScript，另有 Python 基础功能； 你和 Agent 看到的是同一套诊断结果。

  • 实时预览

    直接在 IDE 中查看 Worktree 里的应用；还能在预览画面上 框选元素，把视觉反馈发给 Agent。

  • 跑着也能转向

    Agent 跑到一半，照样能改方向、追加指令，或者一键叫停—— 而且停下就是停下，绝不粉饰成“成功”。

今天就从源码构建

Charter 仍处于开发预览阶段：核心桌面工作流已经可用并通过测试， 安装包、签名和自动更新仍在开发中。目前仅支持从源码构建。

terminal
# Node.js ≥ 22.19、npm 和 Git
$ git clone https://github.com/longyunfeigu/Charter.git
$ cd Charter && npm install
$ npm run dev
  1. 打开一个 Git 项目。Charter 直接处理真实仓库，并把每个任务放进隔离的 Worktree。
  2. 添加模型 Provider。前往设置 → Models，选择 Anthropic、OpenAI、OpenRouter、LiteLLM 或任意兼容端点。密钥会保存在系统钥匙串中。
  3. 创建一个 Session。选择 Agent 和权限模式，说明你想要的结果；任务进入待审查后，再根据证据验收。

想等安装包？可以关注 GitHub Releases；我们正在推进打包发布。

你大概会想问的

为什么还没有安装包？

这是刻意安排的先后顺序。我们先把 Charter 的核心流程——隔离改动、证据、 验证、审查和回滚——做完整、打磨稳定，再处理打包、签名和更新通道。 目前想用 Charter，最直接的办法就是 git clone，所以这里只提供源码构建步骤。

我的代码会被传到哪里？

项目、任务状态和证据账本默认只留在本机。运行 Agent 时， 只有 Prompt 和你附加的上下文会发往你自己配置的模型端点， 不会经过其他后端；数据统计也默认关闭。

能用哪些模型和 Agent？

内置 Charter Agent 支持 Anthropic、OpenAI、OpenRouter、LiteLLM 预设， 也支持自定义 Anthropic 或 OpenAI 兼容端点。已经安装的 Claude Code 和 Codex CLI 也能原生接入，沿用同一套 Session 和审查流程。

和 AI-first IDE 有什么不同？

Charter 不想替换你喜欢的 Agent，而是要替换承载它的那个终端窗口。 AI-first IDE 往往绑定自家的 Agent；Charter 则为你已经在用的 Agent 补上 隔离 Worktree、完整记录、可重跑的验证、默认人工批准的审查关卡， 以及接受后依然可用的回滚。

真的开源吗？

Charter 采用 MIT 许可，并全程公开开发，包括产品规格、 架构决策记录和实现状态。如果页面上的任何说法在仓库里找不到依据，欢迎提 Issue。

Charter 是谁在开发？

Charter 的大部分开发工作由编程 Agent 完成，并且正是按这个页面描述的方式工作： 每次改动都对照公开规格，经过计划、验证和审查。这些关卡之所以存在， 正是因为 Charter 自身的开发也依赖它们。
